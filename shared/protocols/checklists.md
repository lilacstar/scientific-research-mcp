# 内容验证检查清单

本文档为 `verify-content` Skill 的完整检查清单。验证时逐项检查，每项标记为通过/不通过/不适用。

## 一、论证逻辑链检查

### 1.1 Introduction 部分
- [ ] 研究问题是否明确、具体（而非笼统的"研究X"）
- [ ] 研究空白是否有文献支撑（非凭空提出）
- [ ] 研究空白与研究问题之间的逻辑联系是否清晰
- [ ] 本文目的是否直接回应了研究空白

### 1.2 Methods 部分
- [ ] 研究设计是否能够回答 Introduction 提出的问题
- [ ] 变量定义是否清晰、可操作化
- [ ] 样本量是否有估算依据
- [ ] 主要结局指标是否与研究问题对应
- [ ] 是否描述了混杂因素的控制方法
- [ ] 统计方法是否与研究设计匹配

### 1.3 Results 部分
- [ ] 是否呈现了 Methods 中预设的所有主要结局指标
- [ ] 数据是否回答了研究问题
- [ ] 统计结果是否完整（效应量 + 置信区间 + p 值）
- [ ] 是否存在选择性报告（只报告显著结果）

### 1.4 Discussion 部分
- [ ] 是否对所有主要结果进行了解读
- [ ] 是否与已有研究进行了比较
- [ ] 结论是否不超过 Results 证据能支撑的范围
- [ ] 局限性是否完整（不遗漏重要问题）
- [ ] 是否引入了 Results 中没有的新数据或新引用

### 1.5 全文论证链
- [ ] 问题→方案→数据→解释→结论 的完整链条是否无断裂
- [ ] 是否存在循环论证（用结论证明前提）

## 二、内部一致性检查

- [ ] 术语全文一致（同一概念不使用不同名称）
- [ ] 样本量在各章节中一致
- [ ] 研究对象/纳入标准在 Methods 和 Results 中一致
- [ ] 统计方法描述与实际结果中报告的统计量匹配
- [ ] 图表内容与正文描述一致
- [ ] 图表编号按出现顺序且无遗漏
- [ ] 正文引用了所有图表
- [ ] 参考文献：正文中 "X et al." 引用均在文献列表中存在
- [ ] 参考文献：文献列表中的每条文献在正文中都被引用
- [ ] 数字、日期、年份等事实性信息全文一致

## 三、事实性陈述检查

- [ ] 背景描述中的数据有引用支撑（如"据统计，X疾病发病率达Y%"）
- [ ] "首次"、"最大"、"唯一"等绝对性声称有依据
- [ ] 文献引用的结论不歪曲原意
- [ ] 无虚构的参考文献
- [ ] 标注"待人工核查"：所有需要上网验证的具体数字和事实

## 四、常见逻辑谬误检查

- [ ] 未将相关性表述为因果关系（除非研究设计为 RCT）
- [ ] 未存在选择性报告
- [ ] 假设在 Methods 之前已提出（非事后假设）
- [ ] 统计推断无过度解读（避免 p 值边界处不对称解读）
- [ ] 局限性部分未遗漏重要问题

## 五、表述规范性检查

- [ ] 无未引用支撑的断言
- [ ] 无"本研究首次..."等无法验证的声称
- [ ] 无"重大突破"、"革命性"等夸大表述
- [ ] 无"众所周知"、"毋庸置疑"等无引用的绝对化表述
- [ ] Discussion 中无 Results 之外的新数据

## 六、参考文献来源合法性检查

- [ ] 中文文献可在 CNKI / 万方 / 维普中检索到
- [ ] 英文文献可在 Web of Science / Scopus 中检索到
- [ ] 无微信公众号来源的文献
- [ ] 期刊缩写 [J]、学位论文 [D] 等类型标注正确
- [ ] 作者、年份、标题、期刊等信息与实际出版物一致

## 七、内部交叉验证检查（v1.5 新增）

仅比对论文内部文本一致性，不涉及外部数据库验证。输出措辞限于"内部一致"/"内部不一致"。

### 引用格式场景说明（v1.9 新增）

论文引用格式分两种场景，验证时需先识别论文采用的是哪种（或哪种混合），再按对应规则检查：

**场景 A：数字编号引用**（如 `[1]`、`[1,3]`、`[1-5]`）
- 正文以 `[N]` 形式引用，文末参考文献列表以 `[N]` 编号排列
- 此场景需检查：编号对应、编号连续性、编号无重复/乱序
- 常见于中国理工科期刊、医学期刊

**场景 B：作者-年份引用**（如 `（张三，2023）`、`Zhang et al., 2023`）
- 正文以作者名+年份形式引用，文末参考文献列表按作者姓氏字母排序
- 此场景需检查：作者-年份匹配、作者拼写一致、文献列表排序正确
- 常见于社会科学期刊（APA/Harvard 格式）、部分综合性期刊

**场景 C：混合引用**（少见但存在）
- 同时使用 `[N]` 和作者-年份两种格式，需对两种格式分别执行上述检查

### 检查项

- [ ] **引用-文献对应**：正文每个引用标记（无论 `[N]` 还是作者-年份）在参考文献列表中均有对应条目
- [ ] **文献-引用对应**：参考文献列表中的每条文献在正文中至少被引用一次（孤立文献检查）
- [ ] **编号连续性**（场景 A/C 专有）：参考文献编号从 1 开始连续递增，无跳号、无重复
- [ ] **作者-年份匹配**（场景 B/C 专有）：正文引用中的作者名和年份与对应文献列表条目一致
- [ ] **作者拼写一致**：同一作者在正文引用和文献列表中拼写一致（所有场景）
- [ ] **文献列表排序**：数字编号场景按编号递增排列；作者-年份场景按作者姓氏字母排序
- [ ] DOI 字段格式符合 10.XXXX/XXXXX 规范（如存在）
- [ ] 参考文献列表中各条目的年份与标题/期刊信息无矛盾

### 第七组修复建议输出规则（v1.9 新增）

当第七组检查发现不一致时，除标记问题严重程度外，还必须输出结构化修复建议：

**编号跳号/重复/乱序时**，输出建议重排表：
```
建议按正文引用首次出现顺序重新编号：
| 旧编号 | 建议新编号 | 文献标识（作者，年份） |
|--------|-----------|---------------------|
| 3 | 1 | ... |
| 1 | 2 | ... |
```

**正文引用无对应文献列表条目（悬空引用）时**，输出：
```
悬空引用（正文有、列表无）：
- 第 X 处：正文引用 [N] / (Author, Year)，建议补充文献或删除引用
```

**文献列表条目无正文引用（孤立文献）时**，输出：
```
孤立文献（列表有、正文无）：
- [N] Author (Year)：建议在正文中添加引用或从列表中移除
```

**作者-年份不一致时**，输出：
```
作者-年份不一致：
- 正文引用 "(A, Y1)" 与文献列表 [N] "(B, Y2)" 不匹配，建议统一为正确的作者名和年份
```

> 此修复建议与 paper-coordinator v1.7 合并全文时的自动重排功能互补：
> - verify-content 发现问题 → 输出修复建议 → 用户可手动修复或在下次合并全文时用自动重排
> - paper-coordinator 合并时预检 → 发现问题 → 自动重排 → 重排后仍建议执行 verify-content 验证

## 八、期刊要求检查（v1.7 新增，条件执行）

仅当 metadata.json 中 journalConfig 存在且非 null 时执行。缺失或为 null 时全部标记为"不适用"。

- [ ] 全文字数不超过 journalConfig.wordLimit.total（仅当该值 > 0）
- [ ] 引用格式符合 journalConfig.citationFormat 标准（仅当该值非空）
- [ ] 论文包含 journalConfig.sectionRequirements 要求的章节（仅当该值非空）
- [ ] 符合 journalConfig.specialNotes 中的特殊投稿要求（仅当该值非空）

## 检查组与修订类型的映射关系（v1.8 新增）

以下映射用于 verify-content v1.8 的前置快速筛检，确定不同修订类型对应的验证范围。

| 检查组 | 全量 | 核心组（revise） | 增量 | 定向（fix/polish+未关闭） | 跳过（polish） |
|--------|:----:|:----------------:|:----:|:------------------------:|:-------------:|
| 一、论证逻辑链 | ✅ | ✅ | - | 按问题归属 | - |
| 二、内部一致性 | ✅ | ✅ | ✅ | 按问题归属 | - |
| 三、事实性陈述 | ✅ | ✅ | ✅ | 按问题归属 | - |
| 四、常见逻辑谬误 | ✅ | ✅ | - | 按问题归属 | - |
| 五、表述规范性 | ✅ | - | - | 按问题归属 | - |
| 六、参考文献来源 | ✅ | - | - | - | - |
| 七、内部交叉验证 | ✅ | ✅ | ✅ | 按问题归属 | - |
| 八、期刊要求 | ✅（条件） | -（条件） | -（条件） | -（条件） | - |

说明：
- "按问题归属"表示定向验证模式下，仅当上次验证报告在该组发现了问题时才执行
- "条件"表示第八组仅当 journalConfig 存在时执行，无论哪种验证模式
- 第六组（参考文献来源）为高风险检查，在核心组中未包含，但在全量模式下必执行
- 增量验证模式沿用 v1.2 定义（第 2, 3, 7 组），与 v1.8 的新模式并存
- 第七组（内部交叉验证）在核心组（revise）和增量模式中均执行，因为编号一致性问题在内容修订后容易出现（v1.9 调整）
